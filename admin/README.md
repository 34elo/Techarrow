# Панель модерации (Admin)

Веб-приложение для модераторов платформы квестов: очередь заявок на публикацию, одобрение и отклонение с причиной, просмотр и обработка жалоб. Реализовано на **Next.js 16** (App Router), **React 19**, **TanStack Query**, **Tailwind CSS** и **shadcn/ui**.

**Важно:** этот репозиторий содержит **только frontend** админ-панели. Для работы нужен запущенный **backend API** (см. ниже).

## Что нужно для запуска

- **Node.js** 20+
- **pnpm** 10 (удобно через Corepack: `corepack enable` и `corepack prepare pnpm@10 --activate`)
- Рабочий **backend** по адресу, который вы укажете в `NEXT_PUBLIC_API_URL` (по умолчанию ожидается `http://localhost:8000`)

---

## Backend

Админка обращается к REST API (авторизация, модерация квестов, жалобы). Сервер нужно поднять **отдельно** — из репозитория вашего backend-проекта или общей инструкции команды.

- Убедитесь, что API доступен по сети с той машины, где запущен frontend (для Docker на хосте часто нужен не `localhost` внутри контейнера, а IP хоста или `host.docker.internal` — см. вашу ОС и настройки Docker).
- После запуска backend проверьте health/доступность перед входом в панель.

Если frontend и backend на одной машине вне Docker: достаточно `NEXT_PUBLIC_API_URL=http://localhost:8000` (или ваш порт).

## Локальный запуск (без Docker)

```bash
cp .env.example .env
# при необходимости измените NEXT_PUBLIC_API_URL — адрес вашего API

pnpm install
pnpm dev
```

Интерфейс: [http://localhost:3000](http://localhost:3000).

Продакшен-сборка на машине без контейнера:

```bash
pnpm build
pnpm start
```

По умолчанию Next.js слушает порт **3000** (`pnpm start` без дополнительных флагов).

## Запуск в Docker

Переменная `NEXT_PUBLIC_*` встраивается в клиентский бандл на этапе **сборки** образа — задавайте URL API при `docker build` / `docker compose build`, а не только при `run`, если меняете API.

### Production-образ

Сборка и запуск вручную (порт контейнера **3000**, наружу по умолчанию пробрасывается **3001**):

```bash
docker build \
  --build-arg NEXT_PUBLIC_API_URL=http://localhost:8000 \
  -t admin-panel .

docker run --rm -p 3001:3000 admin-panel
```

Через Compose из корня этого репозитория (внешний порт по умолчанию **3001**):

```bash
NEXT_PUBLIC_API_URL=http://localhost:8000 docker compose build
docker compose up
```

Откройте [http://localhost:3001](http://localhost:3001). Порт хоста задаётся переменной `ADMIN_PORT` (по умолчанию `3001`).

### Режим разработки в контейнере

Поднимает `next dev` с монтированием исходников и отдельным volume для `node_modules`:

```bash
NEXT_PUBLIC_API_URL=http://localhost:8000 docker compose -f docker-compose.dev.yml up --build
```

После старта интерфейс на [http://localhost:3001](http://localhost:3001) (или `ADMIN_PORT`, если задан). Для файловых систем вроде Docker Desktop включён polling изменений.

## Демо-аккаунт

Для проверки входа в панель модератора используйте учётные данные, которые должны быть созданы на стороне **backend** и иметь права модератора:

| Поле на форме входа | Значение для демо |
|---------------------|-------------------|
| Email               | `moderator`       |
| Пароль              | `demo123`         |

Если вход не проходит, проверьте: доступность API по `NEXT_PUBLIC_API_URL`, наличие пользователя с этими данными и ролью модератора на сервере, отсутствие блокировок CORS/firewall между браузером и API.

## Публичный деплой

Если приложение развёрнуто публично, добавьте сюда актуальную ссылку:

**URL:** _(укажите после публикации, например `https://…`)_

Убедитесь, что при сборке production-образа или CI в `NEXT_PUBLIC_API_URL` попал адрес того же API, который доступен браузеру пользователя (обычно HTTPS-домен backend, а не внутренний адрес контейнера).
